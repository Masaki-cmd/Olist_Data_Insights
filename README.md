# Olist_Data_Insights
# Olist E-commerce データ分析プロジェクト

## 概要

本プロジェクトは、ブラジルの大手E-commerceプラットフォームOlistの公開データセットを使用し、「収益の最大化」または「コストの最適化」をKGI（最終評価指標）としたデータ分析を行います。分析は「課題定義→仮説→分析→提案」のフローに沿って進められ、最終的には現場で実行可能な具体的なアクションを提案することを目的としています。

## 1. データ読み込みと初期探索

**課題定義**: OlistのE-commerceデータをロードし、その構造、内容、品質を初期的に把握する。

**仮説**: 複数のCSVファイルは互いに関連しており、結合することで注文、商品、顧客、販売者などの総合的なビューが得られる。

**分析**: `archive.zip`を解凍し、以下のCSVファイルをPandas DataFrameとして読み込みました。

*   `olist_customers_dataset.csv`
*   `olist_geolocation_dataset.csv`
*   `olist_order_items_dataset.csv`
*   `olist_order_payments_dataset.csv`
*   `olist_order_reviews_dataset.csv`
*   `olist_orders_dataset.csv`
*   `olist_products_dataset.csv`
*   `olist_sellers_dataset.csv`
*   `product_category_name_translation.csv`

各DataFrameに対し、`head()`, `info()`, `describe()`を実行し、データ概要を把握しました。

**主要な発見と「泥臭さ」**:
*   日付カラムが`object`型であること。
*   `review_comment_title`, `review_comment_message`, および一部の`product`情報に欠損値が存在すること。
*   `product_category_name`にも欠損が存在。

**提案**: 日付型の変換と欠損値の適切な処理が、後の分析の正確性を確保するために必要である。

## 2. データクレンジングと結合

**課題定義**: 複数のデータソースを結合し、一貫性のある分析用データセットを構築する。欠損値や不適切なデータ型を処理し、データ品質を向上させる。

**仮説**: データを結合しクレンジングすることで、各エンティティ間の関係性が明確になり、より深いビジネスインサイトが得られる。

**分析**:
*   **日付カラムの変換**: `order_purchase_timestamp`などの日付関連カラムを`datetime`型に変換しました。
*   **欠損値処理**: `products_df`の`product_category_name`の欠損値を`'unknown'`で補完しました。
*   **カテゴリ名翻訳**: `product_category_name_translation_df`を`products_df`に`left merge`で結合し、ポルトガル語のカテゴリ名を英語に翻訳。未翻訳のカテゴリ名は`'unknown'`で補完しました。
*   **データ結合（Left Merge）**: 以下の順序で`final_df`を構築しました。結合ごとにレコード数を確認し、意図しないデータ欠損や増加がないか検証しました。
    *   `products_df` と `product_category_name_translation_df`
    *   `order_items_df` と `products_df`
    *   `orders_df` と `order_items_products_df`
    *   `orders_items_products_df` と `customers_df`
    *   `orders_customers_df` と `sellers_df`
    *   `orders_customers_sellers_df` と `order_payments_df`
    *   `orders_customers_sellers_payments_df` と `order_reviews_df`

**「泥臭さ」と結合キーの理由**:
*   結合キー (`order_id`, `product_id`, `customer_id`, `seller_id` など) の選定は、各データセット間の論理的な関係性に基づきました。
*   `left merge`を主に使用することで、結合元のDataFrameの情報を最大限保持し、レコードの意図しない削除を避けました。特に`order_payments_df`や`order_reviews_df`との結合では、1つの注文に複数の支払い方法やレビューが存在する可能性があり、レコード数が増加する現象も確認されました。
*   `product_category_name_english`の欠損補完は、翻訳データに存在しないカテゴリが存在するため必要でした。

**提案**: 統合された`final_df`は、ビジネス上の多角的な分析を可能にする基盤となります。

## 3. 収益関連指標の計算と追加データクレンジング

**課題定義**: 分析の核となる収益指標を計算し、残存する数値型データの欠損値を適切に処理する。

**仮説**: `price`と`freight_value`を合算することで、各注文アイテムからの総売上を初期指標として把握できる。物理的特性の欠損値を補完することで、より詳細なコスト分析が可能になる。

**分析**:
*   **`order_item_revenue`の計算**: 各注文アイテムの`price`と`freight_value`を合計し、`order_item_revenue`カラムを作成しました。
*   **製品の物理的特性の欠損値処理**: `product_name_lenght`などの製品の物理的特性に関する欠損値は、まず**カテゴリごと**の中央値で補完し、それでも残った欠損値は**全体**の中央値で補完しました。
*   **データ型の最適化**: `order_item_id`などの整数で扱えるカラムは、`fillna(0).astype(int)`を用いて`int`型に変換し、メモリ効率を向上させました。

**「泥臭さ」**:
*   `order_item_revenue`は現時点では「売上」としていますが、将来的にコストデータが利用可能になれば「利益」へと発展させる余地があります。
*   欠損値の補完において、単純な削除ではなくカテゴリ中央値や全体中央値を用いることで、データ損失を最小限に抑えつつ、分析に必要な情報を保持する判断を行いました。

**提案**: これらの指標とクレンジングされたデータは、収益最大化に向けた具体的な分析の土台となります。

## 4. 初期収益分析と2018年9月以降の売上急減の深掘り

**課題定義**: 月次売上トレンドを分析し、特に2018年9月以降の売上急減の原因を特定する。

**仮説**: 2018年9月以降の売上減少は、データ収集期間の終了、またはビジネス上の重大な問題を示唆している。

**分析**:
*   **主要な収益指標の算出**: 総売上、ユニーク注文数、注文アイテム数、平均単価、平均送料などを算出しました。
*   **月次売上トレンドの可視化**: `order_purchase_timestamp`を基準に月次売上を集計し、折れ線グラフで可視化しました。2018年9月以降の売上急減（10月にはゼロ）が明確に確認されました。
*   **売上急減の深掘り**: 2018年9月以降の注文データを抽出し、`order_status`を分析した結果、ほとんどの注文（20件中19件）が`canceled`（キャンセル）ステータスであることが判明しました。

**主要な決定事項と「泥臭さ」**:
*   2018年9月以降のデータは、Olistのビジネス活動の停止またはデータ収集の終了を示唆すると判断しました。
*   このため、以降の分析対象期間を**2016年9月から2018年8月まで**に限定した`final_df_valid_period`を作成しました。これは、異常データを除外し、有効なビジネス活動期間に基づいた信頼性の高い分析を行うための重要な「泥臭い」データフィルタリングです。

**提案**: 有効期間内のデータに集中することで、実効性のある施策立案が可能になります。

## 5. 有効期間内での収益最大化分析と提案（カテゴリ・商品）

**課題定義**: 有効期間内において、高収益を上げているカテゴリや商品を特定し、収益をさらに伸ばすための具体的な施策を検討する。

**仮説**: 特定のカテゴリや商品が高い平均単価や注文数を誇り、その成功要因を横展開できる。

**分析**:
*   **カテゴリ別収益分析**: `final_df_valid_period`を用いて、`product_category_name_english`ごとの総売上、平均アイテム売上、アイテム数、平均レビュー評価を算出しました。`health_beauty`, `watches_gifts`, `bed_bath_table`などがトップカテゴリであることが判明しました。
*   **トップカテゴリ内の商品深掘り**: 例として`health_beauty`カテゴリ内のトップ10商品を特定し、その売上、価格、送料、レビュー評価を分析しました。

**提案（具体的な施策）**:
1.  **高収益カテゴリの深掘りと成功要因の横展開**:
    *   成功カテゴリ（例: `health_beauty`）の商品特徴、顧客層、マーケティング戦略を詳細に分析。
    *   トップ販売者との連携を強化し、成功事例を他販売者と共有。
    *   他カテゴリへ「商品ページ改善ガイドライン」を提供し、高収益カテゴリのベストプラクティスを適用。
2.  **高単価商品のプロモーション強化とラインナップ拡充**:
    *   「プレミアム商品」特集ページの設置や、バンドル販売（複数商品のセット販売）の促進。
    *   販売者向けに「高単価商品開発支援プログラム」を導入し、魅力的な高単価商品の供給を増やす。
3.  **顧客体験の向上とレビュー評価の改善**:
    *   特に高収益カテゴリにおいて、低評価レビューの内容を深掘りし、具体的な改善策（例: 商品説明の改善、品質管理強化）を講じる。
    *   レビュー促進キャンペーンを実施し、ポジティブなフィードバックを増やす。
    *   品質保証プログラムを導入し、顧客の信頼を向上させる。

## 6. 異なる着眼点での利益向上分析と提案（配送パフォーマンス）

**課題定義**: 配送リードタイムが顧客のレビュー評価に与える影響を分析し、配送効率改善による顧客満足度向上策を検討する。

**仮説**: 配送リードタイムが短いほどレビュー評価が高く、顧客満足度が高い。

**分析**:
*   **配送リードタイムの計算**: `order_approved_at`と`order_delivered_customer_date`が欠損していない注文に絞り込み、両者の差から`delivery_lead_time_days`を計算しました。
*   **可視化と相関分析**: レビュー評価別の配送リードタイムをボックスプロットで可視化し、配送リードタイムとレビュー評価の相関係数（-0.302995）を算出しました。

**主要な発見と「泥臭さ」**:
*   レビュー評価が低いほど配送リードタイムが長く、そのバラつきも大きいことが視覚的に確認されました。
*   配送リードタイムとレビュー評価の間に中程度の負の相関があることが統計的に示され、配送の遅延が顧客満足度を低下させる明確な証拠となりました。
*   配送リードタイムの計算には、関連する日付カラムが完全に揃っているデータのみを使用するという「泥臭い」データ選定を行いました。

**提案（具体的な施策）**:
1.  **配送リードタイムの短縮と顧客体験の向上**:
    *   配送リードタイム監視システムを導入し、遅延が発生しやすい経路や販売者を特定。
    *   販売者に対し、配送パフォーマンス指標を開示し、改善のための具体的な支援（例: 効率的な梱包方法、信頼性の高い配送パートナーの紹介）を行う。
    *   顧客への遅延状況を積極的に共有し、期待値を管理。速達オプションの導入も検討。
2.  **地理情報を用いた物流最適化の検討**:
    *   `geolocation_df`を活用し、販売者と顧客の距離が配送リードタイムに与える影響を分析。
    *   地域別の配送パートナーの評価基準を設け、パフォーマンスの低いパートナーを見直す。
    *   主要な顧客層と販売者層の地理的集中度に基づき、共同配送拠点設置の可能性を検討。

## 今後のタスクと未解決の質問

*   **詳細な利益計算**: 現状の「収益」は`price + freight_value`であり、真の「利益」を算出するためのコストデータ（仕入れコスト、運営コストなど）を組み込む。
*   **残存する欠損値の包括的な処理**: `order_approved_at`, `order_delivered_carrier_date`, `order_delivered_customer_date`などの日付カラム、および`payment_type`, `payment_value`のわずかな`NaN`値について、配送リードタイム分析以外での包括的な処理方針を決定・実行する。
*   **ROIの定量的評価**: 提案された各施策について、具体的なROIや機会費用の定量的評価を行う。
*   **Geolocationデータの本格的な活用**: `geolocation_df`を`final_df_valid_period`に統合し、販売者と顧客の距離計算、地域特性の分析といった本格的な活用を実行する。

これらの未解決の質問やタスクに取り組むことで、より具体的で、定量的根拠に基づいたビジネス提案が可能になります。
